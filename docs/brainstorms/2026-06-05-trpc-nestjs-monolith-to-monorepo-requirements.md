# tRPC + NestJS: учебный проект «монолит → монорепо»

## Summary

Учебный репозиторий из двух частей, демонстрирующий один и тот же домен (Auth / Catalog / Cart) сначала как монолит NestJS с межмодульными вызовами внутри процесса, а затем как монорепо (pnpm workspaces) из трёх отдельных сервисов NestJS, общающихся по tRPC со сквозной типобезопасностью. Документация пошагово разбирает трансформацию и фиксирует best practices.

---

## Problem Frame

Разработчик, знакомый с NestJS, понимает, как строить монолит с модулями, но плохо представляет, что конкретно меняется при разделении на сервисы: где появляется сетевая граница, как сохранить типобезопасность между сервисами и какой ценой. Обычные туториалы по микросервисам либо прыгают сразу к gRPC/брокерам сообщений и теряют наглядность, либо показывают tRPC только в связке «фронтенд → бэкенд», не объясняя, применимо ли это между сервисами.

Момент боли — переход «было/стало»: учащийся видит готовый монолит и готовые микросервисы по отдельности, но не видит сам шов трансформации и не понимает, какие свойства теряются, а какие приобретаются. Из-за этого решения о связанности (coupling) принимаются по аналогии и «как у всех», а не осознанно.

---

## Actors

- A1. Учащийся-читатель: клонирует репо, читает документацию, запускает оба приложения, сравнивает их код бок о бок.
- A2. Сервис Auth: хранит засеянные сессии и пользователей; отвечает на запрос «кто владелец этой сессии».
- A3. Сервис Catalog: хранит каталог товаров и цены; отвечает на запрос деталей товара по id.
- A4. Сервис Cart: публичная точка входа; при добавлении позиции обращается к Auth (владелец сессии) и Catalog (цена/наличие) перед сохранением корзины.

---

## Key Flows

- F1. Добавление товара в корзину — монолит
  - **Trigger:** вызов публичного эндпоинта Cart «добавить позицию» (sessionToken, itemId, qty).
  - **Actors:** A1, A4, A2, A3
  - **Steps:** Cart-модуль получает запрос → вызывает внедрённый AuthService для проверки владельца сессии → вызывает внедрённый CatalogService для получения цены/наличия → собирает строку корзины → сохраняет в in-memory репозиторий Cart.
  - **Outcome:** корзина обновлена; возвращена с актуальной ценой и идентифицированным владельцем. Все вызовы — внутри одного процесса.
  - **Covered by:** R1, R2, R3

- F2. Добавление товара в корзину — монорепо (tRPC)
  - **Trigger:** тот же публичный вызов Cart с теми же аргументами.
  - **Actors:** A1, A4, A2, A3
  - **Steps:** сервис Cart получает запрос → вызывает Auth через tRPC-клиент (`whoami`) по HTTP → вызывает Catalog через tRPC-клиент (`itemById`) по HTTP → собирает строку → сохраняет в свой in-memory репозиторий.
  - **Outcome:** функционально тот же результат, что и в F1; отличается только транспорт (сетевые tRPC-вызовы вместо внедрённых методов). Сигнатуры вызовов совпадают с монолитом.
  - **Covered by:** R5, R6, R7, R8

- F3. Наблюдение потока «из коробки»
  - **Trigger:** A1 запускает приложение по инструкции README и выполняет пример вызова.
  - **Actors:** A1
  - **Steps:** запуск (`монолит`: одно приложение; `монорепо`: три сервиса) → выполнение приведённого в README примера вызова к Cart → наблюдение результата и (в монорепо) фактов межсервисных tRPC-вызовов.
  - **Outcome:** учащийся видит идентичное поведение обоих вариантов без написания собственного клиента.
  - **Covered by:** R4, R9, R11

## Requirements

**Структура репозитория и документация**
- R1. Репозиторий содержит две папки верхнего уровня: `monolith/` и `monorepo/`, каждая со своим README.
- R2. README монолита минимален: назначение проекта и ссылка на проект монорепо (ссылка добавляется/обновляется после публикации на GitHub; до этого — явный плейсхолдер).
- R3. README монорепо подробный: пошаговое объяснение, как монолит был разделён на три сервиса и как организовано межсервисное взаимодействие по tRPC.
- R4. В каждом README есть раздел «как запустить» и хотя бы один пример сквозного вызова к Cart, чтобы поток можно было наблюдать без написания собственного клиента.
- R12. Документация включает раздел best practices, который честно описывает компромисс: tRPC между сервисами даёт связанность на этапе компиляции (compile-time coupling) — уместно в монорепо и неуместно для полиглот- или независимо деплоящихся сервисов (там — gRPC/контракты).
- R13. Документация явно отмечает in-memory хранение как упрощение и указывает замену на Prisma + БД как упражнение «на потом».

**Монолит (NestJS, один процесс)**
- R5. Монолит — обычное приложение NestJS из трёх модулей: Auth, Catalog, Cart.
- R6. Модули общаются между собой через внедрение сервисов (DI), без сети.
- R7. Auth предоставляет проверку владельца сессии по токену на основе засеянных данных; Catalog предоставляет получение товара (цена/наличие) по id; Cart предоставляет добавление позиции и получение корзины.
- R8. Данные каждого модуля хранятся в in-memory репозиториях, засеянных при старте.

**Монорепо (pnpm workspaces, три сервиса, tRPC)**
- R9. Монорепо организовано как pnpm workspaces с тремя сервисами NestJS (Auth, Catalog, Cart) и общим пакетом для базовой настройки tRPC и общих типов.
- R10. Каждый сервис выставляет свой tRPC-роутер и экспортирует его тип; вызывающие сервисы импортируют только тип роутера (`import type`), без рантайм-зависимости от чужого кода.
- R11. Cart — публичная точка входа: его публичный вызов «добавить позицию» внутри обращается к Auth и Catalog через tRPC-клиенты по HTTP.
- R14. Сигнатуры межсервисных вызовов в монорепо совпадают с сигнатурами соответствующих внутримодульных вызовов монолита; единственное отличие — транспорт.
- R15. Конфигурация (порты сервисов, URL соседних сервисов) задаётся через `@itgorillaz/configify`, а не `@nestjs/config`.

**Качество и процесс**
- R16. Разработка ведётся по TDD: тесты пишутся до реализации на каждой итерации.
- R17. ESLint и Prettier подключены с самого начала в обоих проектах; линтер запускается после каждой TDD-итерации и ошибки фиксируются.

## Acceptance Examples

- AE1. **Covers R6, R7.** Дано: монолит запущен, в Auth засеяна сессия `s1` → пользователь `u1`, в Catalog засеян товар `i1` (цена 100, в наличии). Когда A1 вызывает Cart «добавить позицию» с (`s1`, `i1`, qty 2). Тогда корзина пользователя `u1` содержит строку `i1×2` с ценой из Catalog, все вызовы — внутри процесса.
- AE2. **Covers R11, R14.** Дано: монорепо, все три сервиса запущены, те же засеянные данные. Когда A1 вызывает публичный эндпоинт Cart с теми же аргументами. Тогда результат идентичен AE1, при этом Cart выполнил tRPC-вызовы к Auth и Catalog по HTTP, а сигнатуры вызовов совпали с монолитом.
- AE3. **Covers R10.** Дано: исходники монорепо. Когда проверяется код Cart, обращающегося к Auth/Catalog. Тогда из соседних сервисов импортируются только типы роутеров (`import type`), и сборка Cart не тянет рантайм-код Auth/Catalog.
- AE4. **Covers R2.** Дано: проект до публикации на GitHub. Когда читается README монолита. Тогда он содержит назначение проекта и явный плейсхолдер ссылки на монорепо, помеченный как обновляемый после публикации.

## Success Criteria

- Учащийся, склонировав репо, может запустить и монолит, и монорепо по README и выполнить пример вызова к Cart, получив одинаковое поведение.
- Сравнив код двух вариантов, учащийся видит, что место вызова не изменилось по форме — изменился только транспорт, и понимает, почему типобезопасность сохраняется в монорепо (общие типы) и какой ценой (compile-time coupling).
- Downstream-исполнитель (ce-plan / ce-work) может начать реализацию, не изобретая домен, границы сервисов, точку входа или критерии «готово»: всё зафиксировано требованиями и flow.
- Раздел best practices даёт читателю осознанный критерий «когда tRPC между сервисами уместен, а когда нет».

---

## Scope Boundaries

- Настоящая авторизация (JWT, Passport, OAuth, хеширование паролей) — вне рамок; используются засеянные сессии.
- Реальная БД и ORM (Prisma) — вне рамок реализации; упоминаются в документации как следующее упражнение.
- Любой фронтенд / UI — вне рамок; проект только бэкенд.
- Деплой, Docker, Kubernetes, service discovery, балансировка, API-gateway как отдельный сервис — вне рамок.
- Асинхронное взаимодействие через брокеры сообщений (очереди, события) — вне рамок; межсервисные вызовы синхронные по tRPC.
- Завершение покупки/оплата (checkout, платежи) — вне рамок; Cart останавливается на «добавить позицию» / «получить корзину».
- Nx и Turborepo как инструмент монорепо — вне рамок; выбран pnpm workspaces ради прозрачности.

---

## Key Decisions

- Домен Auth / Catalog / Cart: Cart обязан обращаться к Auth и Catalog, поэтому межсервисные вызовы не выглядят надуманными — наглядно демонстрируют ценность tRPC.
- pnpm workspaces (а не Nx/Turborepo): минимум «магии», шов трансформации и совместное использование типов видны напрямую.
- In-memory хранение: ноль настройки, фокус читателя на паттерне tRPC; побочный эффект — каждый сервис владеет своими данными, поэтому Cart физически вынужден ходить по сети.
- Идентичные сигнатуры вызовов в обоих вариантах: превращает «было/стало» в основной обучающий приём; меняется только транспорт.
- Только-типовой импорт роутеров (`import type`): сервисы развязаны в рантайме, но типобезопасны на компиляции — ключевое обоснование монорепо.
- Cart как единственная публичная точка входа: один наблюдаемый сквозной поток вместо REST-контроллеров на каждом сервисе.
- Честный раздел best practices про compile-time coupling: отделяет осознанное применение tRPC от слепого копирования паттерна.

---

## Dependencies / Assumptions

- Предполагается NestJS как базовый фреймворк; при работе подключается скилл `/nestjs-builder` или `/nestjs-expert`.
- Конфигурация — через `@itgorillaz/configify` (скилл `/nestjs-configify`).
- tRPC используется поверх HTTP; типобезопасность между сервисами обеспечивается общими TypeScript-типами в пределах монорепо (предположение: единый язык TypeScript во всех сервисах).
- Сессии и каталог засеяны детерминированно при старте, чтобы примеры в README были воспроизводимы.
- Ссылка из README монолита на монорепо заполняется после публикации на GitHub (до этого — плейсхолдер).
